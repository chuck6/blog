Prometheus联邦配置

# Prometheus联邦配置

prometheus 联邦

# 理论知识

单个Prometheus Server可以轻松处理数以百万级的时间序列，当数据规模扩大时，可以使用联邦集群的方式对Prometheus进行扩展。

对于大部分监控规模而言，只需要在每一个数据中心(例如EC2可用区、Kubernetes集群)安装一个Prometheus Server实例，就可以在各个数据中心处理上千规模的集群，这样单一Prometheus Server对于不同数据中心采集时，避免网络配置的复杂性。

如下面的联邦集群架构所示，在每个数据中心部署单独的Prometheus L用于采集当前数据中心的监控数据。并由一个中心的Prometheus H负责聚合多个数据中心的监控数据。

# 部署配置

# 目标要求

目前的现状是，已有一套Prometheus L1，里面已经针对性的采集了数据库的同步延迟的告警数据信息，目前我们想要通过采集这套Prometheus中的数据到 Prometheus H1中，然后自定义推送告警。

Element是：

otter_delay{delay{instance="192.168.1.100:9508",job="otter_sync",sync_plan_id="1",sync_plan_name="A资源池aaa业务oralce-->B资源池bbb业务mysql",tags_id="11111111111111"}}
1

对应的value值就是相应的同步延迟的时间。

# 联邦配置

针对Prometheus H1 进行配置，在prometheus的配置文件中，增加下面的配置。

scrape_configs:
  - job_name: 'federate_otter_sync'
    scrape_interval: 60s
    honor_labels: true
    metrics_path: '/federate'
    params:
      'match[]':
        - '{job="otter_sync"}'
    static_configs:
      - targets:
        - '1.1.1.1:9090'

  • 这里定义了一个job的名字是 'federate_otter_sync'.
  • scrape_interval: 60s ，设置的是抓取数据的周期，默认是1分钟1次。
  • honor_labels: true ， 如果配置为true的话，可以确保当采集到的监控指标冲突的时候，能够自动忽略冲突的监控数据；如果配置为false, Prometheus 会自动将冲突的标签替换为"exported_" 的形式。
  • match[] 参数指定了 我们需要获取的时间序列，match[] 参数必须是一个瞬时向量选择器。可以配置多个match[]参数，大部分情况下就是匹配原有的prometheus中的job，测试过没办法使用PromQL的方式来过滤指标项，针对性的打上数据标签，只能在后续的rules里面用PromQL过滤指标项，打上告警标签。
  • '1.1.1.1:9090' 这个就是源端的 prometheus的地址。

# 配置告警规则

我们这里需要对 某些特定的tags_id 并且 某些特定的sync_plan_id 来打上告警标签，推送告警，告警阈值是20分钟。

例如我要求 tags_id 是'1111111' 并且sync_plan_id 是"1或者2或者3"的 item的value 大于20分钟时，推送告警。

PromQL的逻辑就是，一个大的PromQL里面内嵌多个小的PromQL，之间是or的关系。内部的PromQL里面又包含2个PromQL，这2个PromQL语句之间是and的关系，也就是tags_id是某个值，sync_plan_id 是可能是多个值。

在alert的rule 的配置中，增加下面的配置项：

   - alert: xx系统otter_delay延迟大于20分钟
     expr: (otter_sync{tags_id="1111111"}  and otter_sync{sync_plan_id=~"1|3|5" } > 20) or (otter_sync{tags_id="2222222"} and otter_sync{sync_plan_id =~"2|4|6" } > 20)
     for: 5m
     labels:
      severity: warning
      app: xx系统
     annotations:
      summary: "检测到otter_delay同步延迟"
      description: '当前延迟时长是{{ printf "%.2f" $value }}分钟)'

同样可以采用上面提及的两种方法，让prometheus对于配置文件生效。

  • alert 描述的是告警规则的名称
  • expr 描述的是使用PromQL表达式完成的告警触发条件，用于计算是否有时间序列满足该条件。
  • for 可选参数，满足告警条件持续多久时间后，才会发送告警。
  • labels 自定义的标签，允许指定额外的标签附加在告警上。
  • annotations 用来设置有关告警的一组信息标签。例如对告警的简要说明文字或相关地址链接，其内容在告警产生时会一起作为参数发送到Alertmanger.
